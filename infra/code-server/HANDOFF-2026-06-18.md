## Confirmed Findings

### 1. "Install code-server on bastion" is not the fix

The page the user is opening is already the bastion-hosted `code-server`. This is not a "container vs bastion host" split in the way the user initially suspected.

Local repo evidence:
- `docker-compose.yml` exposes the browser port and mounts the extensions directory.
- `README.md` already treats this container as the deployment truth source.

Official doc evidence:
- code-server official docs say Docker itself is a supported deployment mode.
- Official FAQ says blank/non-working webviews are commonly caused by insecure browser context, not by Docker alone.

### 2. The environment is currently running in an insecure browser context

Observed directly in the browser:
- Warning banner said `code-server is being accessed in an insecure context. Web views, the clipboard, and other functionality may not work as expected.`

This matches code-server official guidance:
- Webviews depend on service workers.
- Service workers require a secure context.
- Accessing via raw HTTP/IP is a known cause of blank or dead webviews.

This is currently the strongest root-cause candidate for `Continue` and similar extensions rendering `about:blank` or appearing dead.

### 3. Continue had one separate, concrete dependency issue

Browser console showed:
- `Failed to register Continue config.yaml schema, most likely, YAML extension is not installed`

This was fixed remotely by installing the YAML extension inside the running container.

Remote command already run:
```sh
docker exec code-server code-server --install-extension redhat.vscode-yaml
```

Verification already done:
- `redhat.vscode-yaml-1.23.0-universal` now exists in `/root/.local/share/code-server/extensions`

This removes one obvious blocker for Continue, but does not by itself resolve the broader webview problem.

### 4. Cline also shows an extension compatibility problem

Browser console / extension-host logs showed:
- `PendingMigrationError: navigator is now a global in nodejs, please see https://aka.ms/vscode-extensions/navigator`

Interpretation:
- This is not explained by containerization.
- This points to an extension-side compatibility issue with newer VS Code / extension-host behavior.
- Even after secure-context is fixed, `cline` may still require an extension update or downgrade strategy.

### 5. Qoder is not completely dead

Observed behavior:
- `Qoder` UI could open and showed account/login-related UI.

Interpretation:
- The whole extension host is not globally dead.
- Failure mode is mixed:
  - some extensions are blocked by insecure-context webview behavior
  - some have their own compatibility/runtime issues

### 6. TLS verification has been globally weakened in the container

Observed remote environment:
```sh
echo $NODE_TLS_REJECT_UNAUTHORIZED
```

Returned:
```txt
0
```

Extension-host warning also confirmed this in DevTools.

This is not the root cause of blank webviews, but it is operationally important and may affect extension network behavior. It should be reviewed separately.

## Official Documentation Collected

### code-server official docs

Context7 / official code-server docs point to:
- insecure context as the main reason webviews fail
- recommended fixes:
  - access via `localhost` / `127.0.0.1`
  - use a domain with a valid HTTPS certificate
  - use a trusted self-signed certificate
  - temporary browser workaround: treat the origin as secure in Chromium

Important constraint from the user:
- bastion does **not** support local reverse tunnels, so the "just use localhost forwarding" route is not available here

That makes these the realistic next options:
- proper HTTPS on the bastion-facing endpoint
- trusted self-signed certificate if feasible in this environment
- temporary Chrome insecure-origin-as-secure flag only for validation

### VS Code extension docs / migration signal

Context7 lookup confirms the `navigator is now a global in nodejs` message is an extension-compatibility/migration issue on the extension side, not a code-server-container deployment issue by itself.

## Recommended Next Steps

### Priority 1: Prove the secure-context hypothesis quickly

Use one temporary validation path first:
- open the current origin in Chrome with `chrome://flags/#unsafely-treat-insecure-origin-as-secure`

- fully reload code-server

Reason:
- this is the fastest way to separate "webview blocked by insecure context" from "extension itself is broken"

Expected outcomes:
