# 策略效果分析方案

> 来源：数智中台部 · 数据运营组。本 Skill 整合行业方法论与团队实践，仅限集团内部学习使用，请勿外发。

**核心产物**是一份《效果分析方案》：为已上线或即将上线的策略/功能，明确用什么对比基准、怎么归因、要准备哪些数据，把“效果好不好、为什么、接下来怎么办”应如何验证讲清楚。

**附带能力**：当用户已经有运营数据时，可在确认方案后继续执行分析并生成 xlsx 报告。没有数据时，方案本身就是完整交付物，**绝不编造数据**。

**适用场景**：连锁零售各类已上线 / 待上线策略复盘，包括门店运营、营销、商品、数字化工具上线等。只要目标是回答“跟谁比、净效果多少、是否可持续”，就适用。前四种对照方法适配所有场景；**净增量法**用于“策略只覆盖部分对象”的场景，是对主方法的专用补充，不是每次都用。

## 方法选择

每个结论都要回答“跟谁比”。按场景选对比方法：

| 对比类型 | 说明 | 适用 |
|---------|------|------|
| 实验组 vs 对照组 | AB 的直接对比，因果最强 | 有 AB |
| 同比（去年同期） | 消除季节性 | 无 AB，首选 |
| 区域层级对比 | 试点区域 vs 上层整体，扣除大环境趋势 | 无 AB，常用 |
| 匹配对照门店 DID | 给每家试点门店配一家相似门店，按双重差分算净效果 | 无 AB，且有门店级明细 |
| 净增量法（增量来源拆解） | 按覆盖对象的新增 / 持续 / 退出三态拆解效果，扣掉策略内部减量与蚕食 | 策略只覆盖部分对象时 |

- 环比只作辅助参考，不作归因依据。
- 多个方法可叠加互证，但主结论必须有明确主方法。
- 前四种方法主要扣除外部环境影响；净增量法主要拆解策略内部增减量。
- 混淆排查、负面指标分析是检查项，不与归因方法并列。

## 两条路径

| 场景 | 归因难度 | 方法 |
|------|---------|------|
| 有 AB 实验 | 较低 | 实验组 vs 对照组 |
| 无 AB（全量 / 分批上线） | 较高 | 同比 + 区域对比 + 匹配 DID（如条件满足）+ 混淆排查 |

公司门店按 `大区 → 省区 → 营运区 → 片区 → 门店` 分层管理，区域对比建立在这套层级上（细节见 [references/methods.md](references/methods.md)）。

## 执行步骤

### 输出前硬规则

任何场景先过以下硬规则：

| 规则 | 要求 |
|------|------|
| 先定路径 | 明确标注“有 AB”或“无 AB”，不要混用两条路径的主结论 |
| 先出方案 | 用户没有提供可计算数据时，只交付方案和取数模板，不生成结论数字 |
| 不编数据 | 没有明细、没有基线、没有对照值时，用待填项，不用经验数替代 |
| 主护栏同口径 | 护栏指标按主指标同样的对比方法跑一遍，不能只看主指标 |
| 结论带可信度 | 无 AB 场景必须给高 / 中 / 低可信度，并说明限制来源 |
| 无数据必带最小取数清单 | 用户只有方案需求、暂无可计算数据或数据尚未准备好时，交付物必须包含独立章节 `最小取数清单`（允许别名 `最低取数清单`），不能只在正文里零散提数据准备 |

### 场景判别

先判断交付形态：

| 用户信息 | 交付形态 | 方法边界 |
|---------|----------|----------|
| 有 AB，且已有实验结果 | 效果分析方案 + 可继续执行结果分析 | 主结论只用实验组 vs 对照组；同比只作佐证 |
| 有 AB，但没给实验数据 | AB 结果分析取数模板 | 不判断显著性，不给推广结论 |
| 无 AB，已全量上线 | 无 AB 效果分析方案 | 默认同比 + 区域层级；有门店明细才加 DID |
| 即将上线，尚无数据 | 上线后效果分析方案 | 只定义基准、指标、取数表、干扰排查 |
| 策略只覆盖部分对象 | 在对应主方法外追加净增量法 | 增删类三态拆解；调结构类扣同类蚕食 |

### 第一步：收集分析信息

向用户确认以下信息（未提供则主动询问）：

| 必需信息 | 说明 |
|---------|------|
| 策略 / 功能名称 | 分析什么 |
| 策略内容 | 具体做了什么改变 |
| 上线时间和范围 | 何时上线、覆盖哪些区域 / 门店 / 对象 |
| 是否有 AB 实验 | 有 → 第四步-A；无 → 第四步-B |
| 同期事件 | 上线期间还发生了什么 |
| 汇报对象 | 决定详细程度和表述 |
| 核心 + 护栏指标 | 主指标与护栏指标名称、口径、优先级 |

> 拿到背景信息后先出方案，不要急着要明细数据。方案确认后、且用户确有数据时，再进入执行分析。

### 第二步：输出《效果分析方案》

先出方案，让用户看到“怎么分析、要准备哪些数据”。

**方法要精准、可落地。** 按场景选最合适的方法，每个方法都要落到一张用户可直接取数填数的表；方案交付的是**取数模板**，不是泛泛的方法说明。

选方法的默认原则：

- 有 AB：实验组 vs 对照组，一个方法即可。
- 无 AB：默认“同比 + 区域层级对比”。
- 用户明确要求更精确、且能拿到门店级明细：再追加匹配对照门店 DID。
- 策略只覆盖部分对象：追加净增量法。
- 混淆排查：只列用户真正提到的同期事件。

方案必须讲清三件事：

1. **分析方法**：选什么方法，以及为什么适用。
2. **对比模板**：每个方法配一张可填数表。行是逐个指标，列是各对比组 / 时段原始值、净效果公式、判定列。
3. **数据与干扰**：列出要准备的数据、粒度、时间范围，以及同期干扰与处理方式。

模板骨架：

*对照组 / 匹配 DID*

| 指标 | 口径·单位 | A 试点·前 | B 试点·后 | C 对照·前 | D 对照·后 | 净效果 =(B−A)−(D−C) | 判定 |
|------|----------|:--:|:--:|:--:|:--:|:--:|------|
| 〈逐个核心指标与护栏指标，每行带单位〉 |  |  |  |  |  |  |  |

*同比 + 区域层级*

| 指标 | 口径·单位 | 试点·今年 | 试点·去年同期 | ① 试点同比 | 上层·今年 | 上层·去年同期 | ② 上层同比 | 净效果 = ①−② | 判定 |
|------|----------|:--:|:--:|:--:|:--:|:--:|:--:|:--:|------|
| 〈逐个核心指标与护栏指标，每行带单位〉 |  |  |  |  |  |  |  |  |  |

*净增量法（增删类策略）*

| 对象状态 | 口径·单位 | 取数期 | 产出 | 归类 |
|---------|----------|:--:|:--:|:--:|
| 新增对象 | 〈按对象取，商品=元/店/天〉 | 覆盖后 |  | 毛增量（+） |
| 持续对象 | 同上 | 后 − 前 |  | 保留变化（±） |
| 退出对象 | 同上 | 覆盖前 |  | 减益（−） |
| 净增量 |  | — |  | 毛增量 + 保留变化 − 减益 |

> 三态加总 = 对象总体净变化。主推 / 陈列 / 资源倾斜等“对象不变、只调内部权重”的策略，改用“对象总体是否真增”口径扣同类蚕食（见 [references/methods.md](references/methods.md)）。

### 方案型交付的最小取数清单

当用户暂无可计算数据、只要求“先给方案”，或数据尚未准备好时，交付物中必须包含独立章节 **`最小取数清单`**（允许别名 **`最低取数清单`**）。

该章节必须使用下表结构，列名固定，不改写、不删列：

| 表名 | 粒度 | 字段 | 用途 |
|------|------|------|------|
| 〈示例：门店日销表〉 | 〈示例：门店-天〉 | 〈示例：门店ID、日期、销售额、客流〉 | 〈用于同比 / 区域对比 / DID〉 |

使用规则：

- 该表用于列出**执行已批准分析方法所需的最低数据集合**。
- 只写最低必需项，不展开成全量数仓需求清单。
- 若一个方法当前因数据不足不可执行，也应在此表中体现其最低依赖字段，方便业务 / 数据团队直接取数。
- 这是无数据或暂未拿到数据场景的默认输出要求，不能仅用 prose 提及“后续补数据”。

### 第三步：确认方案，询问是否有运营数据

1. 请用户审核方案：方法是否合适，混淆因素是否遗漏，数据粒度能否拿到。
2. 询问是否有运营数据：
   - **有** → 进入第四步执行分析、出报告。
   - **无** → 方案即本次交付物，用户先取数，后续再回来执行分析。

### 第四步-A（有 AB 实验）：实验组 vs 对照组

**核心对比**：

| 指标 | 对照组 | 实验组 | 差异 | p 值 | 是否显著 |
|------|--------|--------|------|------|---------|
| 主指标 / 辅助指标 / 护栏指标 | - | - | - | - | - |

**细分对比**：按区域层级 / 门店特征 / 品类 / 时间拆解，重点检查：

- 辛普森悖论
- 哪些细分效果最好 / 最差
- 若策略只覆盖部分对象，补做净增量拆解

**结论与建议**：给出是否显著优于旧方案、效果幅度、推广建议、风险提示。

### 第四步-B（无 AB 实验）：同比 + 区域对比

无对照组，靠多维度对比构建“准对照”。执行时把数据逐指标填入第二步的表，回填净效果与判定列；DID 计算与显著性检验交给脚本，不手算。

> 护栏指标也要一并同比 / 对比。主指标提升必须建立在护栏不破的前提下。

**① 同比**：策略覆盖范围指标对去年同期。若去年同期有特殊事件，需剔除或标注。

**② 区域层级对比**：用试点区域上一层级整体作大环境基准，`净效果 ≈ 试点变化 − 上层整体变化`。

**③ 匹配对照门店 DID**：只有在能拿到门店级明细时才做。

```bash
python3 scripts/did_analysis.py 数据.json
python3 scripts/did_analysis.py 数据.json --json
```

- 前提是平行趋势。
- 门店对数少时，不强求显著性，主看同向比例 + 净效果幅度。

**④ 混淆因素排查**：只列用户提到的同期事件。

| 混淆因素 | 时间 | 可能影响 | 排除方法 | 结论 |
|---------|------|---------|---------|------|
| （如促销活动） | - | - | 对比非促销品类同比变化 | - |

**⑤ 净增量法 / 增量来源拆解**：策略只覆盖部分对象时使用。

- **增删类**：`净增量 = 毛增量 + 保留变化 − 减益`。
- **调结构类**：`净增量 = 毛增量 − 蚕食`，用“对象总体是否真增”兜底。

详见 [references/methods.md](references/methods.md) 第五节。

**⑥ 负面指标分析**：若指标恶化，先同比确认，再做区域对比与维度拆解，最后做 tradeoff 评估。

### 归因可信度（无 AB 必做）

| 可信度 | 条件 |
|-------|------|
| 高 | 同比 + 区域对比 + DID 三者方向一致且数值接近，混淆因素已排除 |
| 中 | 同比和区域对比方向一致，但 DID 不可行或平行趋势未过；有少量混淆因素无法精确排除 |
| 低 | 仅有同比，缺区域对比和 DID，多个混淆因素交织 |

可信度只能到“低”时，要坦诚说明，并建议下次同类策略提前用 `ottos-retail-ab-test` 设计 AB。

### 第五步：输出自检清单

| 检查项 | 通过条件 |
|--------|---------|
| 有对比基准 | 每个结论都说清“跟谁比” |
| 有反事实 | 回答了“不做这个策略会怎样” |
| 平行趋势已检验 | 用 DID 时已检验历史趋势平行 |
| 混淆已排查 | 列出同期事件并逐一排除 |
| 细分与总体一致 | 检查了分区域 / 分品类与总体是否一致 |
| 区分短期长期 | 分析了可持续性 |
| 负面指标已分析 | 恶化指标有对比数据和根因 |
| 净增量已拆 | 覆盖部分对象时，已从增益中扣出减量减益 |
| 可信度已标注 | 明确标注可信度等级 |
| 最小取数清单已提供 | 无数据 / 方案型交付时，已提供独立 `最小取数清单` 章节，且表头为 `表名 | 粒度 | 字段 | 用途` |

### 第六步：生成 xlsx 报告

用 `scripts/build_report.py` 出报告，不手写 openpyxl 样板：

```bash
python3 scripts/build_report.py config.json [输出.xlsx]
python3 scripts/build_report.py
```

配置结构、sheet 组织、公式校验见 [references/xlsx-report.md](references/xlsx-report.md)。DID 结果可直接使用 `did_analysis.py --json` 输出填充。

## 输出格式

- **核心产物：《效果分析方案》**（Markdown，每次必出）
- **附带产物：效果分析报告**（xlsx，仅在有运营数据时才出）

**文档头**（Markdown 方案、xlsx Sheet1 都带）：

| 项 | 值 |
|----|----|
| 策略名称 | - |
| 日期 | 当天（YYYY-MM-DD） |
| 版本 | v0.x 草稿 / v1.0 初稿 / v2.0 正式 |

**命名**：方案 `[策略名称]-效果分析方案.md`，报告 `[策略名称]-效果分析报告.xlsx`。

**数字规范**：

- 占比 / 比率型变化用 `pp`
- 数值 / 金额型用 `%` 或绝对值，不混用
- 金额带单位（万元 / 亿元）
- md 用 `+/-` 加粗，xlsx 用绿 / 红字体区分正负

**文体要求**：

- 方案是交付文档，用书面陈述。
- 章节标题一律用名词短语，不用动词短语，不用对话口吻。
- 同一份方案只设一个“待确认事项”区，统一放末尾。
- 正文客观陈述方法与判断，避免口语化提醒。

**模板要求**：

- 分析方法章节以填数表为主体。
- 每个方法一张可填数表，判定 / 可信度列在方案阶段留空，执行后回填。
- 无数据或仅方案场景，必须单独提供 `最小取数清单` / `最低取数清单` 章节，且表结构固定为：`表名 | 粒度 | 字段 | 用途`。

## 跟上游工具的衔接

> 本节仅用于提示后续步骤，不要自动调用其他 skill。

- **指标口径沿用上游**：主指标、护栏指标沿用 `ottos-north-star-metric` 定义。
- **有 AB 最直接**：若策略曾用 `ottos-retail-ab-test` 设计实验，直接走第四步-A。
- **ROI 换算**：把净效果代入上游效益换算公式，得到实际 ROI。
- **可信度低就反推上游**：事后归因可信度低时，建议下次提前设计 AB。
