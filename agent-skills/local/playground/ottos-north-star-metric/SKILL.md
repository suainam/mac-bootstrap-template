# 北极星指标与护栏指标设计

> 来源：数智中台部 · 数据运营组。本 Skill 整合行业方法论与团队实践，仅限集团内部学习使用，请勿外发。

为产品或策略设计完整的指标衡量体系。

## 核心概念

- **北极星指标（North Star Metric）**：衡量产品/策略在当前阶段是否成功的唯一核心指标（不同阶段关注重点不同，北极星也随之变）。需满足：可归因于产品本身、能反映用户价值、可量化可追踪
- **核心驱动指标**：把北极星拆成几个关键子指标，定位"是哪一环在带动它"，用于归因和早期判断方向
- **护栏指标（Guardrail Metrics）**：追求北极星过程中不能破的底线，防止顾此失彼
- **效益换算公式**：把北极星的变化量翻译成利润增量的公式。北极星回答"做得好不好"，效益换算公式回答"这个好值多少钱"。提前定义可避免结项时争论收益怎么算，也能反向检验北极星是否选对——换算不出经济价值的北极星，说明离业务目标太远

## 执行步骤

### 输出前硬规则

指标设计先过 6 条硬规则，避免产物完整但口径失真：

| 规则 | 要求 |
|------|------|
| 北极星只有 1 个 | 不用多个主指标并列替代取舍 |
| 先确认策略路径 | 不知道产品/策略具体改变哪个环节时，先给待确认项，不硬选指标 |
| 后移一层价值 | 推广数、开通数、下发数等过程指标只能作驱动指标，北极星必须落到业务价值结果 |
| 护栏有阈值 | 护栏必须写底线值或待确认阈值，不能只写方向 |
| 效益落利润 | ROI 分子用利润增量或成本节省，不用销售额直接替代 |
| 口径可复用 | 北极星、护栏、效益公式要能直接交给 AB 实验和效果复盘沿用 |

### 场景判别

先按业务场景决定指标重心：

| 场景 | 北极星候选方向 | 必设护栏 |
|------|---------------|----------|
| 商品 / 组货 / 选品 | 采纳后动销、毛利、库存效率的净改善 | 关联品类销售、毛利率、缺货率 |
| 门店运营 / 陈列 / 动线 | 店效、坪效、米效、转化或连带的净改善 | 客诉、缺货、执行成本、毛利率 |
| 营销 / 促销 / 会员 | 增量利润、复购、转化或客单的净改善 | 毛利率、补贴成本、非活动期回落 |
| 内部系统 / 数字化工具 | 实际使用后带来的业务指标或效率改善 | 执行合规、人工负担、异常率 |
| 流程提效 / 自动化 | 节省人时、减少错误、缩短周期 | 质量、合规、用户体验 |

### 第一步：收集业务信息

向用户确认以下信息（如用户未提供，主动询问）：

| 必需信息 | 说明 |
|---------|------|
| 产品/策略名称 | 为什么东西设计指标 |
| 业务目标 | 要解决什么问题 |
| 计划的策略路径 | 打算优化哪个环节来达成目标（如"提升电商销售"是改页面 UI 还是改支付流程）。路径决定指标的可操作性与后续归因 |
| 业务阶段 | 冷启动？增长期？成熟期？不同阶段北极星不同 |
| 用户/对象 | 谁在用这个产品，谁受策略影响 |
| 业务方的"既要又要" | 业务方提了哪些期望（往往互相矛盾，这些就是护栏指标的来源） |
| 已有指标 | 现在在看什么指标，有哪些不满意 |

### 第二步：设计指标体系

#### 2.1 北极星指标（1 个）

| 项目 | 内容 |
|------|------|
| 指标名称 | - |
| 定义 | 一句话说清楚衡量的是什么 |
| 计算口径 | 分子/分母/时间窗口/数据来源 |
| 为什么选这个 | 可归因性，可操作性，以及它对应的用户价值时刻（参考下方选择标准） |
| 当前基线值 | 定稿前用上述口径拉取后回填——基线能区分好坏、有提升空间才说明指标选对，同时是设定目标值和后续效果对比的起点 |
| 目标值 | 基于业务预期的目标范围 |

**选择标准**：
- 可归因性：指标变化能归因于产品/策略本身，而非外部因素（季节、促销等）
- 可操作性：团队能通过产品改进影响这个指标
- 用户价值：能反映最终用户获得的价值

**警惕"虚荣指标"**：

虚荣指标是那种数字看着漂亮、但不反映真实价值的指标，比如纯注册数、PV、点击数、累计下载量。它们容易涨（靠推广、靠强制开通就能拉上去），但涨了不代表产品成功，不能做北极星。

⚠️ **特别注意"过程指标"陷阱**：对 ToB / 内部系统来说，"推广门店数""接入业务方数量"这类**不是虚荣指标**（业务方愿意接入本身就是认可），但它们是**过程指标**——只能说明"东西被铺出去了"，不能说明"东西真的产生价值了"。北极星要选"过程指标后面那一层"：

| 过程指标（不要选作北极星） | 后面那一层（北极星该选的） |
|------|------|
| 推广覆盖门店数 | 这些门店实际采纳推荐后的业务指标变化 |
| 系统注册/开通账号数 | 实际活跃使用、且产生效果的比例 |
| 推荐目录下发数 | 目录被采纳后带来的动销率/毛利提升 |

判断方法：问一句"这个指标涨了，能不能说明用户/业务真的获得了价值？"——如果只能说明"铺开了"但说不清"有用了"，那它是过程指标，要往后再追一层。

#### 2.2 核心驱动指标（按可拆解的环节数定，通常 1-3 个）

| 驱动指标 | 定义 | 对应北极星的哪一环 | 负责方 |
|---------|------|------------------|--------|
| - | - | 对应哪个策略动作/环节 | 谁来关注和优化 |

**设计思路**：
- 拆解原则：每个驱动指标对应北极星的一个环节（通常对应第一步的某个策略路径），合起来能解释北极星怎么变
- 选可操作的：团队动作能直接撬动，既是过程监控点也是归因中间项
- 数量克制：按能拆出的有意义环节定，1-3 个即可，不为凑数硬加

#### 2.3 护栏指标（2-3 个）

| 护栏指标 | 定义 | 底线值 | 为什么是护栏 |
|---------|------|--------|------------|
| - | - | 不能突破的阈值 | 追求北极星时最可能牺牲什么 |

**设计思路**：
- 问自己：如果只追求北极星，最可能牺牲什么？那个东西就是护栏
- 常见方向：用户体验、成本/效率、合规/风险
- **替代效应提醒**：如果策略针对部分商品/品类优化，必须把"关联品类的销售变化"设为护栏——优化 A 可能挤掉 B 的销售，整体未必变好

#### 2.4 效益换算公式

效益换算公式最终要落到**利润增量**（而非销售额增量），因为 ROI 的分子是利润。

| 项目 | 内容 |
|------|------|
| 效益类型 | 增收 / 降本 / 提效（选择主要类型） |
| 换算公式 | 北极星变化量如何折算为**利润增量**（写出具体公式） |
| 关键假设 | 公式依赖的假设条件（如毛利率、人天单价等），使用保守估算 |
| 归因方式 | 如何确认利润增量是本项目带来的，而非其他因素 |

**换算公式示例**：

| 北极星指标变化 | 效益换算公式 | 效益类型 |
|--------------|------------|---------|
| 动销率提升 X pp | 新增动销 SKU 数 × 平均 SKU 日均销售额 × 毛利率 × 天数 | 增收 |
| 库存周转天数下降 X 天 | 平均库存金额 × X / 365 × 资金成本率 | 降本 |
| 人工操作耗时减少 X 分钟/店 | X / 60 × 门店数 × 工作日 × 时薪 | 提效 |

**试算验证**：定义公式和参数后，用脚本代入计算得出具体数字，避免多步乘法的手算错误。脚本输出每步中间结果和最终金额。

**避免高估**：公式参数应使用保守值而非平均值。例如"新增动销 SKU"是原来卖不动的商品，动销后销售额通常远低于全店平均，建议取平均值的 30%-50%。如有 AB 实验数据，更稳健的做法是直接用实验组 vs 对照组的总利润差值。

**归因方式**（按严谨程度排列）：
- **AB 实验增量法**：有对照组时，差值即项目增量，天然排除外部干扰
- **反事实推算**：全量上线无对照组时，用历史数据建模预测"不做会怎样"，差值即项目贡献
- **前后对比 + 修正**：上线前后直接对比，手动剔除已知混杂因素，归因最弱但最常用

#### 2.5 指标关联关系

说明指标之间的逻辑关系：
- 哪些驱动指标之间是正相关的
- 哪些指标之间存在天然冲突（取舍）
- 护栏指标在什么情况下会被触发

### 第三步：输出自检清单

| 检查项 | 通过条件 |
|--------|---------|
| 北极星可归因 | 指标变化能说明是产品做得好，而非外部因素的功劳 |
| 北极星非虚荣/非过程指标 | 不是注册数/PV 这种虚荣指标，也不是"推广覆盖数"这种过程指标，而是反映真实价值的"后面那一层" |
| 驱动指标能解释北极星 | 如有驱动指标，几个合起来能说明北极星怎么变，且团队动作能直接撬动 |
| 护栏真正约束 | 护栏是追求北极星时最可能牺牲的东西，而非随意选的 |
| 无互斥遗漏 | 不可能同时提升的指标对已识别，并说明了取舍 |
| 口径明确 | 每个指标的计算方式无歧义，不同人算出来结果一致 |
| 可获取 | 所需数据在现有系统中可获取 |
| 效益可换算 | 北极星变化能通过公式换算为利润增量，假设条件已与业务方对齐 |
| 归因方式明确 | 已确定归因方式（AB 实验 / 反事实 / 前后对比） |
| 效益数字合理 | 利润增量占相关业务总利润的比例合理，参数使用了保守估算 |

### 第四步：给出迭代建议

基于业务场景，给出 2-3 个迭代方向：
- 随着业务阶段变化，北极星可能需要怎么调整
- 哪些驱动指标未来可以进一步拆解
- 护栏指标的阈值怎么根据实际数据修正

### 第五步：提示下游衔接

> 本节是**提示用户后续步骤**用的：指标设计完成后，在输出末尾简短提及即可，**不要自动调用其他 skill**。后续工具由用户按需另行发起。

指标体系是策略闭环的起点，输出末尾提醒用户后续两步怎么接：

- **验证阶段**：策略上线前，这套指标（尤其北极星和护栏）直接拿去做 AB 实验设计——用 `ottos-retail-ab-test`，北极星作主指标、护栏作护栏指标，省去重新定指标。
- **复盘阶段**：策略上线后，用 `ottos-effect-analysis` 做效果分析，沿用同一套指标口径，对比"实际达成 vs 当初定的目标值"。
- **效益闭环**：这里定义的效益换算公式，到结项时把实际数据代入，就是创新项目申报要的 ROI，不用临时凑。

三个工具用同一套指标口径，才能串成"定目标 → 验方案 → 看效果"的闭环。

## 输出格式

**输出方式**：默认在当前项目目录生成一份 Markdown 文档，同时把内容在聊天窗口同步输出。Markdown 文档作为正式产品文档，用于项目立项、目标确认后归档进项目资料；聊天窗口同步展示，方便用户多轮对话修改。文件默认命名 `[产品名称]-指标体系设计.md`。

**文档头**（文档开头固定加）：

| 项 | 值 |
|----|----|
| 项目名称 | - |
| 日期 | 当天（YYYY-MM-DD） |
| 版本 | v0.x 草稿 / v1.0 初稿 / v2.0 正式 |

**固定输出结构**（写进文档的正式内容，每次保持一致；迭代建议、下游衔接只在对话提、不入文档）：

1. 一句话结论（这套指标体系的核心选择）
2. 北极星指标（表）
3. 核心驱动指标（表，如有则出）
4. 护栏指标（表）
5. 效益换算公式（表）
6. 指标关联关系
7. 自检清单及通过状态

**其他**：

- 指标定义一律用表格呈现，便于直接引用
- 百分点用 `pp`、百分比用 `%`，不要混用；效益金额带单位（万元/亿元）

**文体**：输出的是正式产品文档，用书面陈述。章节标题用名词短语，不用动词短语或对话口吻（如「护栏指标」不写成「怎么定护栏」）；正文客观陈述，避免「请你确认 / 你要…」这类对话口吻。

## 注意事项

- 不要堆砌指标，北极星只要 1 个，护栏不超过 3 个
- 护栏指标要有明确的底线值，而非只给方向
- 如果存在明显的指标冲突，主动指出并建议取舍策略
- 指标口径要具体到计算公式级别，避免模糊描述
